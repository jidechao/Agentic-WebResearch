# ADR-0007: 当前日期集中注入（LLM seam 层）

- 状态：已接受
- 日期：2026-07-19

## 背景

发现一个**致命**的时效性幻觉：问"北京今天天气如何？"，模型把"今天"答成
**2025 年 7 月 25 日**（实际是 2026-07-19），并把幻觉的年份带进搜索词（"北京今天天气 2025年7月"）
和答案框架。

根因：**所有 prompt 都没有任何当前时间信号**。模型受训练截止影响，默认活在 ~2025，
对"今天/今年/最新/最近"这类词只能猜，猜错的日期又二次污染下游。

波及面不止 ReAct 执行器：

| 环节 | 受影响 |
|---|---|
| ReAct 执行器 | "今天天气""最新股价"——直接答错日期 |
| Planner | "调研**今年** X"→ 子任务可能被拆成"调研 2025 年…" |
| 子任务执行器 | 搜索词带错年份，召回偏旧 |
| 报告生成 | "截至今年"框架错 |

机器时钟是准的（`datetime.now()` 取到 2026-07-19），问题只是模型拿不到这个信息。

## 决策

**在 LLM 调用 seam（`agent/llm.py`）集中注入当前日期**——延续 [[0001-unify-llm-reply]]
的 seam 思路：把"告诉模型今天日期"这个横切关注点封在统一入口里。

- `_today_line()`：`datetime.now()` → `当前日期：2026年7月19日（星期日）。`（中文 + 星期）。
- `_inject_date(messages)`：消息列表头部注入——**已有 system 消息则合并进去**（前置日期行），
  **否则插一个 system 消息**；不产生双 system 消息。
- `chat` / `chat_stream` 在构造 kwargs 前各调一次 `messages = _inject_date(messages)`。

一处改动，路由 / 规划 / 重规划 / 补研 / 执行器 / Critic / 报告 Reviewer / 报告生成
**全部环节自动带上"今天"**，未来新增 prompt 无需手动加、永不漏。

## 为什么集中而非逐 prompt

| 维度 | 集中（llm.py，采纳） | 逐 prompt（拒绝） |
|---|---|---|
| 覆盖面 | 全部环节 | 仅时效类（planner + 执行器 + 报告 ≈ 4-5 处） |
| 未来新增 prompt | 自动带上 | 需手动记得加，易漏 |
| 改动点 | 1 处 | 4-5 处 |
| 多余 token | 路由/Critic 这类不需日期的调用多 ~15 token | 零 |
| 风险 | 低（seam 层、测试 mock chat 不受影响） | 中（漏一处就复发） |

~15 token/调用的代价可忽略；防漏与可维护性的收益压倒性。

## 未采纳项（防重提）

- **import 期常量**（`TODAY = date.today()` 在 config）：REPL 跨午夜会 stale（差 1 天也比差 1 年好，
  但没必要）。采纳的是**调用期**计算（`datetime.now()` 在每次 chat 取），永远新鲜。
- **只注入到执行器 system prompt**：能修好"今天天气"，但漏掉 planner 对"今年"的解析。
  覆盖不全，否决。

## 验证

- 离线：`test_fixes_unit.py` D1——`_inject_date` 合并/插入/格式三态断言；`test_dag_unit.py` 未回归。
- 真实 API：重跑 `北京今天天气如何？`，确认报告的"今天"为 **2026-07-19**（而非幻觉的 2025）。

## 与 dag_store "不直接调时间函数" 的关系

[[0003-deferred-candidates]] 记录 `make_run_id` 的 timestamp **由调用方传入**、引擎内部不调时间，
是为了 run_id 的**可测/可复现**。本 ADR 在 `chat` 调 `datetime.now()` 不冲突——`chat` 本身
就是**非确定性**的外部调用（调 LLM），注入运行期日期正是其职责，不涉及确定性/可复现性。
