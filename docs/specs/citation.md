# Citation（引用溯源）规格

> 配套决策追溯见 ADR-0009。

## 问题陈述

调研报告当前全部是自然语言文本——所有"来源"只靠模型在子任务总结时自觉保留。用户阅读最终报告时，无法从某条断言跳转到支撑它的证据来源。模型可能丢链接、改写链接、甚至凭记忆编造链接。结果是：报告看起来言之有物，但可信度无法验证，溯源成本完全转嫁给读者。

## 解决方案

为最终报告引入结构化引用机制：正文内联 [N] 编号标记每条断言的来源，末尾附参考列表把每个编号展开为「标题 · URL · 日期」。证据 ID（s1/s2/...）由联网搜索工具在拿到博查 API 结果时分配并写入全局证据池，URL 与标题直接取自 API 返回（零幻觉）。报告生成后，由后处理统一把运行期的 [sN] 锚点重映射为连续的 [1]..[n] 展示编号，并自动拼接参考列表。

## 用户故事

1. 作为报告读者，我希望报告正文的每条关键断言旁有编号标记，这样我能知道它声称的依据是什么。
2. 作为报告读者，我希望报告末尾有完整的参考列表，这样我能逐条核对来源。
3. 作为报告读者，我希望点开参考列表里的链接就能到达原始网页，这样我能自行判断证据可信度。
4. 作为报告读者，我希望参考列表显示来源的发布日期，这样我能判断信息的时效性。
5. 作为报告读者，我希望正文标记和末尾列表的编号一一对应，这样我对照时不会混淆。
6. 作为项目维护者，我希望引用 URL 来自搜索 API 而非模型自报，这样报告不会出现编造的死链。
7. 作为项目维护者，我希望引用 ID 在整个任务运行期间唯一且稳定，这样并发执行的子任务不会产生编号碰撞。
8. 作为项目维护者，我希望同一个 URL 被多次搜到时只算一条引用，这样参考列表不会出现重复条目。
9. 作为项目维护者，我希望中断恢复后引用功能仍然可用，这样历史报告的参考列表不会丢失。
10. 作为项目维护者，我希望模型偶尔标注了不存在的引用编号时系统自动剔除，这样报告不会出现指向虚无的标记。
11. 作为项目维护者，我希望参考列表只包含正文实际引用过的来源，这样列表与正文紧密对应而非罗列全部搜索结果。
12. 作为项目维护者，我希望 ReAct 路径（简单任务）和 Plan-Execute 路径（DAG/线性）都支持引用，这样无论走哪条路径报告都可溯源。
13. 作为项目维护者，我希望现有 ReAct 循环与子任务执行器的函数签名尽量不变，这样改动范围可控、回归风险低。

## 实现决策

- **新增证据模块（evidence）**：负责全局证据池、URL 规范化去重、ID 分配、报告后处理。单一职责，不依赖 LLM 与网络。

- **证据条目 schema**：每条记录为 {id, title, url, date}。id 形如 s1、s2；title/url/date 取自博查 API 的 name / url / datePublished 或 dateLastCrawled。不存储搜索摘要正文（摘要已在 ReAct 消息中被模型消费，进池纯属冗余）。

- **全局证据池（单 run 作用域）**：整个任务共享一个池与一个递增计数器。计数器的原子递增由 CPython GIL 保证；但"查去重表未命中 → 分配新 ID → 写入"是 check-then-set 序列，不在单条字节码内——并发下两个子任务同搜一个 URL 时可能各自分到不同 ID、致去重失效（正是用户故事 #8 要防的）。因此 add_evidence 用一把模块级互斥锁包住整个查询-分配-写入序列，彻底兑现"同 URL 只算一条"的去重保证。池写入是 append-only。

- **ID 分配锚定在工具层**：证据 ID 由联网搜索工具在解析博查返回时分配，而非由 LLM 在写总结时自报。这是本规格最核心的决策——参考列表条目的 URL 由证据池（API 直取）唯一决定；报告后处理只按 [sN] 查池展开，模型无法向参考列表注入自编 URL，从源头杜绝幻觉链接。（模型在 ReAct 消息里仍会看到工具返回的 URL 文本，但它无法影响参考列表的最终 URL——后者由池唯一决定。）

- **URL 规范化去重**：分配 ID 前对 URL 做规范化（去除 fragment、去除 utm_ 等 tracking 参数、host 小写）。规范化后命中已存在 URL 则复用首次分配的 ID，不新增条目。不同 URL 即便指向同一事实也不去重（不同来源，溯源价值不同）。

- **报告后处理流程**：反思循环结束后、落盘前，单遍扫描正文中的 [sN] 标记，按首次出现顺序建立 {sN: k} 映射；池中不存在的标记（幻影引用，源于模型幻觉）从正文删除；随后正则替换所有 [sN] 为连续的 [k]（k 从 1 起）；最后在正文末尾拼接参考章节，按新编号列出，格式为「[k] [标题](URL) · 日期」（markdown 可点链接，满足用户故事 #3）。只有正文实际引用过的来源进入参考列表。**执行时机**：在报告级反思循环全部结束后、落盘前执行一次。反思循环每一轮的 review 看到的是原始 [sN] 标记——"证据充分性"维度可直接评判模型是否标注了引用（标得差即触发重写），循环天然纠偏；若每轮都重映射，模型基于已重映射文本继续写会导致编号错乱。

- **运行期 ID 与展示 ID 分离**：模型在子任务执行期间见到的 [sN] 是运行期 ID，全局唯一但不连续；连续的 [1]..[n] 仅在报告最终展示时由后处理生成。两层各管各的：运行期 ID 稳定保证模型不会编号错乱，展示 ID 连续保证阅读美观。

- **生命周期**：新任务从三个入口之一发起时重置证据池（ReAct 路径入口、Plan-Execute 线性引擎入口、Plan-Execute DAG 引擎入口）；DAG 引擎的 resume 恢复入口不 reset，而是从落盘文件 load 回填（见下条持久化）。

- **持久化与中断恢复**：DAG 引擎将证据池落盘，与现有计划/状态/报告同目录、同原子写模式。/resume 时回填池并将计数器起点设为已加载条目数加一，保证恢复后新分配的 ID 与历史不碰撞。ReAct 路径与线性引擎不涉及恢复，无需落盘。

- **提示词约束强化**：子任务系统提示要求"引用证据时必须使用搜索结果里出现的 [sN] 标记，不要自编编号"；报告汇总提示要求"保留各子任务结果里的 [sN] 标记，系统会自动展开为参考列表"。

- **现有循环签名零改动**：ReAct 循环深模块、子任务执行器、反思闭环的函数签名全部不变。证据池作为模块级全局状态，联网搜索工具直接读写，调用方无感知。

## 测试决策

- **测试哲学**：只测外部行为（纯逻辑的输入-输出），不测实现细节。优先复用现有测试模式。

- **被测模块**：新增证据模块的纯逻辑部分——URL 规范化、ID 分配与去重、池重置与回填、报告后处理（重映射与幻影删除）。

- **既有先例**：现有 DAG 离线单测——模块级函数 + assert + __main__ runner 的离线单测模式，mock LLM，不打真实 API。本规格新增测试完全对齐此模式。

- **新增测试接缝（单一）**：一个测试文件，覆盖证据模块全部纯逻辑分支——URL 规范化去重（同一 URL 不同 tracking 参数返回同 ID）、ID 全局递增与 reset 重开、带历史数据 reset 后计数器起点正确、报告后处理的重映射连续性、幻影引用删除、未被引用的池条目不进参考列表。

- **不测的部分**：提示词是否被模型遵守、真实博查结果是否正确入池——属于 LLM 与网络行为，单测无法覆盖，沿用现有冒烟测试的真实 API 模式兜底，不在本规格范围。

## 范围之外

- Critic 与报告级 Reviewer 不新增 citation 评审维度。现有"证据充分性"维度已间接约束引用质量；schema 不再膨胀。若实测发现模型普遍不标引用，再行评估。
- 不新增 config.toml 配置项（如引用开关、自定义参考列表标题）。内联标记加末尾列表是固定行为，无旋钮可调。
- 不对子任务级中间产出单独做参考列表。引用溯源只面向最终报告。
- 不实现引用强度分级（如"强证据/弱证据"）、不实现引用的语义类型标注。
- 不实现导出为 BibTeX/RIS 等外部文献格式。

## 进一步说明

- 本规格的决策经由一次结构化需求澄清（七个问题逐一确认）得出，核心权衡见 ADR-0009。
- 关键架构决策：引用的真实性锚定在联网搜索工具层（API 直取 URL），而非让 LLM 自报——这是"零幻觉链接"的唯一保证。
- 并发安全：ID 分配的计数器递增依赖 CPython GIL 下 itertools.count.__next__ 的原子性；去重的查询-分配-写入序列由模块级互斥锁保护（见实现决策"全局证据池"）。若未来迁移到自由线程构建或多进程执行，需复核并扩展锁的范围。
- 无标记兜底：若模型全程未标注任何 [sN]（无视 prompt 约束），finalize_report 产出的报告将无参考列表——这是诚实降级，不强制补造引用。
